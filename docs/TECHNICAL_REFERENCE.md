# WSMS PRO v4.3 - Technical Reference

## Architecture

WSMS PRO is generated by installer scripts and centered around:

- `~/scripts/wsms-config.sh` (single source of runtime configuration)
- Runtime modules in `~/scripts/`
- Structured logs in `~/logs/wsms/`

Technical notes:

- For full runtime availability of all commands listed by `wp-help`, install ClamAV packages used by WSMS security modules.
- WSMS is an author-driven personal server management system and includes solutions used in the author's own environment.

## Installers

- `installers/install_wsms.sh` (English)
- `installers/install_wsms_pl.sh` (Polish)

Both installers provide:

- live terminal plus file logging,
- detailed trap-based error diagnostics,
- generated script deployment,
- Bash and Fish alias setup,
- cron provisioning.

## Runtime Module Groups

1. Observability: health audit, fleet status (with SSL certificate expiry monitoring), multi-instance audit.
2. Maintenance: automated update engine (all sites, single site, single plugin, single theme), permissions orchestrator.
3. Backup: lite/full/mysql/interactive/red-robin.
4. Retention and rollback: retention manager (standard clean, force-clean, emergency per-site, emergency-global), rollback engine.
5. Security and sync: ClamAV scans, NAS sync.
6. Local DNS mapping: hosts synchronization (`wp-hosts-sync`) generated by installer.
7. Alerting: `wsms-notify.sh` (email alert module), `wsms-daily-check.sh` (daily cron health check).

## Fleet Status — SSL Monitoring


`wp-fleet-status-monitor.sh` checks the SSL certificate expiry for each managed site using `openssl s_client` and `date -d`. The domain is resolved dynamically:

- For each entry in `SITES`, the script runs `wp option get home` (as the correct user) to extract the real domain from the WordPress configuration (home URL). If this fails, it falls back to the SITES nickname.
- This ensures SSL expiry is checked for the actual domain in use, not just the config label.

Each site row in fleet output includes:

- `SSL: N d` (EN) / `SSL: N dni` (PL) — days until expiry, green if ≥ 14 days, red if < 14 days.
- `SSL: N/A` (yellow) — certificate unreachable or parse error.

Note: `date -d` (GNU coreutils) is required; this is available on Linux servers but not macOS.

## Hosts Synchronization

`wp-hosts-sync` reads domains from `SITES` in `~/scripts/wsms-config.sh` and maintains a marker-based block in `/etc/hosts`:

- start marker: `# >>> WSMS LOCAL HOSTS >>>`
- end marker: `# <<< WSMS LOCAL HOSTS <<<`

The operation is idempotent and preserves non-WSMS lines in `/etc/hosts`.

## Uninstaller Behavior

`tools/wsms-uninstall.sh` supports:

- standard mode,
- force mode (`--force`),
- preview mode (`--dry-run`).

Uninstall removes WSMS shell blocks from `~/.bashrc`, `~/.bash_profile`, and `~/.config/fish/config.fish`. Sed patterns are version-agnostic and match any `WSMS PRO vX.Y` marker, ensuring stale blocks from older installations are also removed. Also cleans the WSMS marker block from `/etc/hosts`.

In addition to marker-based cleanup, uninstaller also removes legacy v4.2 shell blocks that were written without start/end markers:

- Fish header pattern: `# WSMS PRO vX.Y - FISH SHELL ALIASES`
- Bash header pattern: `# WSMS PRO vX.Y - BASH SHELL ALIASES`

For these legacy blocks, the script removes the block body and old helper echo lines (`wp-help`, `wp-status`, `wp-health`) while preserving non-WSMS user lines around the block.

A dedicated regression test validates this behavior in an isolated HOME fixture:

- `tests/test_uninstaller_legacy_cleanup.sh`

## ACL and wp-config.php Permissions

`infrastructure-permission-orchestrator.sh` applies `setfacl -R -m u:$USER:r-x` to site directories to grant the deployment user read+execute access for backup operations. Because the recursive flag sets `r-x` on all files including `wp-config.php`, the ACL for that file is immediately overridden to `r--` after the recursive pass. This ensures `stat` reports `640` (not `650`) and `wp-audit` does not raise a false security alert.

## Email Alerting

WSMS includes a lightweight email alert system based on two modules:

- `wsms-notify.sh` — provides `send_alert()` function, sourced by other scripts.
- `wsms-daily-check.sh` — runs three diagnostic scripts and sends a consolidated report.

Configuration in `~/scripts/wsms-config.sh`:

```bash
ALERT_EMAIL=""           # leave empty to disable all alerts
ALERT_ON_FAILURE="yes"   # send alert on critical failure
ALERT_ON_SUCCESS="no"    # send alert on successful cron run
```

`send_alert()` silently exits if `ALERT_EMAIL` is empty — no configuration required to disable.

**Alerts triggered by:**

| Script | Trigger |
|---|---|
| `wsms-daily-check.sh` | daily cron: web server down, disk full, WP unreachable |
| `wp-automated-maintenance-engine.sh` | rollback triggered after failed update |
| `wp-smart-retention-manager.sh` | disk usage exceeds `DISK_ALERT_THRESHOLD` |
| `server-health-audit.sh` | web server down or disk critical |
| `clamav-auto-scan.sh` | infected files found |
| `clamav-full-scan.sh` | infected files found |

**Daily cron setup** (runs at 07:00):

```bash
0 7 * * * bash $HOME/scripts/wsms-daily-check.sh
```

Requires `mail` (e.g. `bsd-mailx`) and a configured MTA (e.g. `postfix`, `msmtp`) on the server.

For the simplest Ubuntu relay setup with `msmtp`, see `docs/MAIL_CONFIGURATION.md`.

## Logging

Logs are categorized under `~/logs/wsms/`:

- `maintenance/`
- `backups/`
- `security/`
- `sync/`
- `retention/`
- `rollback/`

## Repository Note

Repository `scripts/` is not the runtime source of truth.
Runtime modules are generated by installers into user home `~/scripts/`.

## Runtime Preview Exporter

For repository-side inspection, `tools/wsms-export-runtime-scripts.sh` parses installer deploy heredocs and exports module previews into:

- `scripts/runtime-preview/en/`
- `scripts/runtime-preview/pl/`

Preview files are git-ignored. Generate them locally when needed for review or audit:

```bash
bash tools/wsms-export-runtime-scripts.sh
# or selectively:
bash tools/wsms-export-runtime-scripts.sh --only wp-automated-maintenance-engine.sh,wp-smart-retention-manager.sh
```

The installer remains the only source of truth. Preview files are local-only artifacts.

> **Note:** `wsms-config.sh` is intentionally excluded from the export. It is generated directly by the installer (`cat >`) during installation and contains user-specific values (hosts, paths, SSH keys). It is not a `deploy` heredoc block and is not part of the runtime module set.
