# Contributing to WSMS PRO

Thanks for contributing.

## Development Workflow

1. Fork and clone the repository.
2. Create a branch: `git checkout -b feature/my-change`.
3. Implement and test your changes.
4. Open a pull request with a clear summary.

## Validation

Run before opening PR:

```bash
# Full test suite (syntax, required files, behavioral checks)
bash tests/test_suite.sh

# Optional shellcheck (if installed)
shellcheck installers/*.sh tools/*.sh tests/*.sh || true
```

## CI Troubleshooting

If tests pass locally but fail in GitHub Actions CI, the most common cause is
`scripts/runtime-preview/` not existing in CI — this directory is `.gitignored`
(generated artifacts) and must be created by the CI pipeline step before tests run.

**Quick diagnosis:**

```bash
# Check if runtime-preview files are tracked by git (should return 0 if .gitignored)
git ls-tree -r HEAD -- scripts/runtime-preview/ | wc -l
# if 0 — files are not in the repo (expected — they are generated)

# Confirm they are excluded by .gitignore
git check-ignore -v scripts/runtime-preview/en/some-script.sh
# if it prints a line — the path is .gitignored (expected)
```

The CI workflow runs `bash tools/wsms-export-runtime-scripts.sh` before
`bash tests/test_suite.sh` to generate those files from the installers.
If you add a new script to the installers, verify the export step produces it:

```bash
bash tools/wsms-export-runtime-scripts.sh
ls scripts/runtime-preview/en/
```

## Runtime Script Rule

Runtime scripts are generated by installers into `~/scripts/`.
Do not maintain production runtime modules in repository `scripts/`.

When adding a new module:

1. Add generation block in `installers/install_wsms.sh`.
2. Add equivalent generation block in `installers/install_wsms_pl.sh`.
3. Add or update aliases in installer alias phases.
4. Update docs (`README.md` and `docs/TECHNICAL_REFERENCE.md`).

## Logging Rule

New generated scripts should use WSMS logging conventions and write to `~/logs/wsms/...` categories.
