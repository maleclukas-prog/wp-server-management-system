# Contributing to WSMS PRO

Thanks for contributing.

## Development Workflow

1. Fork and clone the repository.
2. Create a branch: `git checkout -b feature/my-change`.
3. Implement and test your changes.
4. Open a pull request with a clear summary.

## Validation

Run before opening PR:

```bash
# Syntax checks
for f in installers/*.sh tools/*.sh tests/*.sh; do bash -n "$f" || exit 1; done

# Optional shellcheck (if installed)
shellcheck installers/*.sh tools/*.sh tests/*.sh || true

# Repository checks
bash tests/test_suite.sh
```

## Runtime Script Rule

Runtime scripts are generated by installers into `~/scripts/`.
Do not maintain production runtime modules in repository `scripts/`.

When adding a new module:

1. Add generation block in `installers/install_wsms.sh`.
2. Add equivalent generation block in `installers/install_wsms_pl.sh`.
3. Add or update aliases in installer alias phases.
4. Update docs (`README.md` and `docs/TECHNICAL_REFERENCE.md`).

## Logging Rule

New generated scripts should use WSMS logging conventions and write to `~/logs/wsms/...` categories.
